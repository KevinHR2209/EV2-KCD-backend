🔧 Cambios realizados — detalle técnico

1

Refactorización del docker-compose.yml

docker-compose.yml
FIX
🎯 Justificación
El archivo original tenía las credenciales de base de datos escritas directamente en el YAML (ej: MYSQL_ROOT_PASSWORD: root123). Esto significa que cualquier persona con acceso al repositorio puede ver las contraseñas. La solución estándar de la industria es usar variables de entorno inyectadas desde un archivo .env que nunca se sube al repositorio. Además, se agregó un healthcheck a MySQL para que Spring Boot no intente conectarse antes de que la base de datos esté lista (este era el motivo principal de los errores de arranque).
❌ Antes
# Problema: credenciales visibles y sin healthcheck
  mysql:
    environment:
      MYSQL_ROOT_PASSWORD: root123
      MYSQL_DATABASE: despacho_db
  # Sin healthcheck → Spring Boot falla al conectar
  ms-ventas:
    environment:
      DB_PASSWORD: root123  # hardcodeado
✅ Después
# Solución: variables desde .env + healthcheck en MySQL
  mysql:
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}  # viene del .env
      MYSQL_DATABASE: despacho_db
    volumes:
      - mysql_data:/var/lib/mysql          # persistencia
      - ./mysql-init:/docker-entrypoint-initdb.d  # crea ventas_db
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5
  ms-ventas:
    environment:
      DB_PASSWORD: ${DB_PASSWORD}  # no expuesto
    depends_on:
      mysql:
        condition: service_healthy  # espera a MySQL
📌 Conceptos aplicados
  • Separación de configuración y código (12-Factor App, Factor III)
  • Healthcheck: garantiza orden de arranque correcto entre contenedores dependientes
  • Named volume (mysql_data): los datos persisten aunque se detenga el contenedor
  • Build args en el frontend: las URLs del backend se inyectan en tiempo de compilación de Vite
2

Creación del script de inicialización de MySQL

mysql-init/init.sql
NUEVO
🎯 Justificación
MySQL tiene una funcionalidad oficial: cualquier archivo .sql colocado en /docker-entrypoint-initdb.d/ se ejecuta automáticamente la primera vez que se crea la base de datos. Se aprovecha este mecanismo para crear ambas bases de datos (despacho_db y ventas_db) sin intervención manual. Sin este archivo, ms-ventas lanzaba un error al intentar conectarse a ventas_db que no existía.
✅ Contenido del archivo
CREATE DATABASE IF NOT EXISTS despacho_db
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

CREATE DATABASE IF NOT EXISTS ventas_db
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
📌 Por qué utf8mb4

Es el encoding recomendado en MySQL 8 porque soporta el conjunto completo de Unicode incluyendo emojis y caracteres especiales del español (tildes, ñ). utf8 clásico solo cubre 3 bytes y puede causar errores con caracteres especiales.

3

Creación del archivo de variables de entorno plantilla

.env.example
NUEVO
🎯 Justificación
El archivo .env real nunca debe subirse a GitHub porque contiene contraseñas. Sin embargo, los colaboradores necesitan saber qué variables deben configurar. La solución estándar es mantener un .env.example con los nombres de todas las variables y valores de ejemplo seguros. Cada desarrollador copia este archivo como .env y completa sus propios valores.
✅ Contenido
# MySQL
MYSQL_ROOT_PASSWORD=root123
DB_USERNAME=root
DB_PASSWORD=root123

# Frontend (URLs del backend)
# LOCAL: usar localhost
# PRODUCCION: usar IP publica de EC2 backend
VITE_API_DESPACHO=http://localhost:8081
VITE_API_VENTAS=http://localhost:8082
📌 Flujo de trabajo
  • .env.example → se sube a GitHub (valores de ejemplo)
  • .env → se agrega a .gitignore, NUNCA se sube
  • En producción: las variables se inyectan como GitHub Secrets

Creación del pipeline CI/CD con GitHub Actions

.github/workflows/deploy.yml
FEAT
🎯 Justificación
La pauta EP2 exige un pipeline que se active automáticamente con push a la rama deploy. El pipeline elimina la necesidad de conectarse manualmente a cada servidor para desplegar: al hacer git push origin deploy, GitHub Actions hace el build, sube las imágenes a Docker Hub y despliega en ambas EC2 automáticamente.
✅ Etapas del pipeline
#EtapaQué hace
1CheckoutDescarga el código del repositorio en el runner
2Login Docker HubSe autentica con el usuario/token de Docker Hub
3Build + Push FrontendCompila React+Vite con las URLs del backend e inyecta ARGs
4Build + Push ms-despachoCompila el JAR con Maven y genera imagen Alpine
5Build + Push ms-ventasCompila el JAR con Maven y genera imagen Alpine
6Deploy EC2 BackendSSH a EC2 privada → git pull → crea .env → docker compose up
7Deploy EC2 FrontendSSH a EC2 pública → git pull → crea .env → docker compose up

Creación de la rama deploy en GitHub

git branch deploy
FEAT
🎯 Justificación
El pipeline deploy.yml está configurado con el trigger on: push: branches: [deploy]. Esto significa que el pipeline solo se activa cuando hay un push a la rama deploy, nunca en main. Esta es una práctica estándar de GitFlow: main es el código estable, deploy es la rama que gatilla el despliegue a producción.
✅ Comandos ejecutados
git checkout -b deploy
git push -u origin deploy
# Resultado: rama deploy creada localmente y en GitHub
📌 Flujo de trabajo con las ramas
  • Desarrollas en main (o ramas de feature)
  • Cuando quieres desplegar: git push origin main:deploy
  • GitHub Actions detecta el push y ejecuta el pipeline automáticamente
  • En ~5 minutos la nueva versión está en producción en AWS
